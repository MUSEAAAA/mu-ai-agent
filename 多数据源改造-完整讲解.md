# 多数据源改造完整讲解

## 一、背景与目标

### 项目结构

```
mu-ai-agent
 ├── JdbcChatMemory    → 聊天记忆持久化（存对话历史）
 ├── PgVectorStore     → RAG 向量库（存知识文档向量）
 ├── DashScope LLM     → AI 模型（通义千问）
 └── 一个数据源         → 阿里云 PostgreSQL
```

### 问题

聊天记忆和 RAG 向量库共用**同一个数据源**（阿里云 PG），你的目标是：

```
聊天记忆 → 本地 MySQL
RAG 向量库 → 阿里云 PostgreSQL（保持不变）
```

---

## 二、核心方案：多数据源

**一句话**：一个应用里注册两个 `DataSource` bean，各连各的库：

```
cloudDataSource  → 阿里云 PG    → cloudJdbcTemplate  → PgVectorStore（RAG）
localDataSource  → 本地 MySQL   → localJdbcTemplate  → JdbcChatMemory（聊天记忆）
```

### 涉及的关键 Spring 概念

| 概念 | 作用 |
|------|------|
| `@Configuration` | 标记这是一个配置类，Spring 会扫描并执行里面的 `@Bean` 方法 |
| `@Bean` | 声明一个由 Spring 容器管理的对象（bean） |
| `@Primary` | 当有多个同类型 bean 时，标记哪个是默认注入的 |
| `@Qualifier("beanName")` | 显式指定要注入哪个 bean |
| `@ConfigurationProperties(prefix = "...")` | 把 yml 里指定前缀的配置绑定到 Java 对象上 |
| `DataSourceProperties` | Spring Boot 内置类，负责读取和翻译数据库连接配置 |

## 三、配置详解

### 第一步：application.yml — 配置双数据源

```yaml
spring:
  datasource:
    cloud:                                     # 云端 PG（RAG）
      url: jdbc:postgresql://你的地址/库名
      username: 用户名
      password: 密码
      driver-class-name: org.postgresql.Driver
    local:                                     # 本地 MySQL（聊天记忆）
      url: jdbc:mysql://localhost:3306/db08?...
      username: root
      password: 1234
      driver-class-name: com.mysql.cj.jdbc.Driver
```

`spring.datasource.cloud` 和 `spring.datasource.local` 是两个独立的前缀，分别描述两个不同的数据库连接。

### 第二步：DataSourceConfig.java — 注册双数据源

完整代码如下：

```java
@Configuration
public class DataSourceConfig {

    // ========== 云端 PG（标记 @Primary）==========

    @Primary
    @Bean(name = "cloudDataSourceProperties")
    @ConfigurationProperties(prefix = "spring.datasource.cloud")
    public DataSourceProperties cloudDataSourceProperties() {
        return new DataSourceProperties();
    }

    @Primary
    @Bean(name = "cloudDataSource")
    public HikariDataSource cloudDataSource(
            @Qualifier("cloudDataSourceProperties") DataSourceProperties properties) {
        return properties.initializeDataSourceBuilder()
                .type(HikariDataSource.class)
                .build();
    }

    @Primary
    @Bean(name = "cloudJdbcTemplate")
    public JdbcTemplate cloudJdbcTemplate(@Qualifier("cloudDataSource") DataSource ds) {
        return new JdbcTemplate(ds);
    }

    // ========== 本地 MySQL ==========

    @Bean(name = "localDataSourceProperties")
    @ConfigurationProperties(prefix = "spring.datasource.local")
    public DataSourceProperties localDataSourceProperties() {
        return new DataSourceProperties();
    }

    @Bean(name = "localDataSource")
    public HikariDataSource localDataSource(
            @Qualifier("localDataSourceProperties") DataSourceProperties properties) {
        return properties.initializeDataSourceBuilder()
                .type(HikariDataSource.class)
                .build();
    }

    @Bean(name = "localJdbcTemplate")
    public JdbcTemplate localJdbcTemplate(@Qualifier("localDataSource") DataSource ds) {
        return new JdbcTemplate(ds);
    }
}
```

**关键设计：**

1. **两条链路完全对称**：`Properties → DataSource → JdbcTemplate`，各自独立
2. **cloud 全部标记 @Primary**：已有代码不传 `@Qualifier` 时默认走云端，不影响旧逻辑
3. **用 `DataSourceProperties` 做中间层**：为什么必须这样？见下文

### 第三步：JdbcChatMemory — 注入本地 JdbcTemplate

```java
@Component
public class JdbcChatMemory implements ChatMemory {

    private final JdbcTemplate localJdbcTemplate;

    // @Qualifier 指定要 local 的，不走 @Primary 的 cloud
    public JdbcChatMemory(@Qualifier("localJdbcTemplate") JdbcTemplate localJdbcTemplate) {
        this.localJdbcTemplate = localJdbcTemplate;
    }
}
```

### 第四步：PgVectorVectorStoreConfig — 注入云端 JdbcTemplate

```java
@Bean
public VectorStore pgVectorVectorStore(
        @Qualifier("cloudJdbcTemplate") JdbcTemplate cloudJdbcTemplate,
        EmbeddingModel dashscopeEmbeddingModel) {
    return PgVectorStore.builder(cloudJdbcTemplate, dashscopeEmbeddingModel)
            .dimensions(1536)
            .distanceType(COSINE_DISTANCE)
            .indexType(HNSW)
            .initializeSchema(true)
            .build();
}
```

## 四、为什么 DataSourceProperties 是必要的？

这个问题很多人都会踩坑，值得仔细讲。

### 直觉写法（错误）

```java
@Bean
@ConfigurationProperties(prefix = "spring.datasource.cloud")
public DataSource dataSource() {
    return DataSourceBuilder.create().build();
}
```

### 发生了什么？

1. `DataSourceBuilder.create().build()` → 创建 `HikariDataSource` 对象
2. `@ConfigurationProperties` 把 yml 里的 `url` 绑定到这个对象上
3. HikariDataSource 的连接地址属性叫 **`jdbcUrl`**（不是 `url`）
4. Spring Boot 对**默认前缀 `spring.datasource`** 有特殊处理：通过 `DataSourceProperties` 做了一层 `url → jdbcUrl` 的翻译
5. 但你用的是**自定义前缀** `spring.datasource.cloud`，这个翻译没了
6. `url` → `setUrl()` 被调用，但 HikariCP 没有 `setUrl()`，只有 `setJdbcUrl()`
7. 最终 `jdbcUrl = null` → 报错

### 修正写法（正确）

```java
@Bean(name = "cloudDataSourceProperties")
@ConfigurationProperties(prefix = "spring.datasource.cloud")
public DataSourceProperties cloudDataSourceProperties() {
    return new DataSourceProperties();       // ← DataSourceProperties 有 setUrl()
}

@Bean
public HikariDataSource cloudDataSource(
        @Qualifier("cloudDataSourceProperties") DataSourceProperties properties) {
    return properties.initializeDataSourceBuilder()   // ← 内部做 url → jdbcUrl 翻译
            .type(HikariDataSource.class)
            .build();
}
```

**DataSourceProperties 的作用：**
- 它的 setter 就叫 `setUrl()` → 完美匹配 yml 的写法
- `initializeDataSourceBuilder()` 内部做了 `url → jdbcUrl` 翻译，再传给 HikariCP

### 对比

| 写法 | 绑定目标 | `url` 能否匹配 | 结果 |
|------|---------|---------------|------|
| 直接绑 HikariDataSource | HikariDataSource.setJdbcUrl() | ❌ | jdbcUrl is required |
| 通过 DataSourceProperties | DataSourceProperties.setUrl() | ✅ | 正常启动 |

## 五、问题排查方法：堆栈阅读

这是整个过程中你遇到的典型报错，学会看它：

```
Caused by: BeanCreationException: Error creating bean with name 'pgVectorVectorStore'
  Caused by: BeanCreationException: Error creating bean with name 'cloudDataSource'
    Caused by: BeanInstantiationException: Failed to instantiate [HikariDataSource]
      Caused by: DataSourceBeanCreationException: Failed to determine a suitable driver class
        at DataSourceProperties.determineDriverClassName(DataSourceProperties.java:176)
        at DataSourceProperties.initializeDataSourceBuilder(DataSourceProperties.java:125)
        at DataSourceConfig.cloudDataSource(DataSourceConfig.java:29)
```

### 阅读方法

**从最底层 Caused by 向上读，只看 at 行：**

1. **最底层** `DataSourceConfig:29` 调了 `initializeDataSourceBuilder()`
2. 它调了 `determineDriverClassName()`
3. 发现 `driverClassName = null` 且 `url = null` → 推导不出驱动 → 抛异常

**再问：为什么 url 是 null？**

`@ConfigurationProperties(prefix = "spring.datasource.cloud")` 去 yml 找 `spring.datasource.cloud.url`。如果 yml 里写的是 `cloude`（多了一个 e），路径不匹配，所有属性都绑定不上。

**一个 typo，整个链都崩了。**

## 六、完整架构图

```
                    application-local.yml
                   /                    \
    cloud.url=xxx...                    local.url=xxx...
                   |                          |
    @ConfigurationProperties(cloud)    @ConfigurationProperties(local)
                   |                          |
        DataSourceProperties           DataSourceProperties
           .initializeDataSourceBuilder()
                   |                          |
        cloudDataSource (Hikari)      localDataSource (Hikari)
                   |                          |
        cloudJdbcTemplate             localJdbcTemplate
                   |                          |
        PgVectorStore (RAG)          JdbcChatMemory (聊天记忆)
                   |                          |
        阿里云 PostgreSQL              本地 MySQL
```

两个数据库、两个 DataSource、两条 SQL 执行链路，互不干扰。需要哪个就在 `@Bean` / `@Autowired` 上用 `@Qualifier` 指定。

## 七、核心知识点速查

| 知识点 | 一句话 |
|--------|--------|
| 多数据源的本质 | 注册多个 `DataSource` bean，各有各的连接配置 |
| @Qualifier | 告诉 Spring "我要这个名字的 bean，不要默认的 @Primary 那个" |
| DataSourceProperties | Spring Boot 内置的数据库配置翻译器，把 `url` 转成 HikariCP 认的 `jdbcUrl` |
| @ConfigurationProperties | yml 配置到 Java 对象的映射器，按前缀匹配 |
| 堆栈阅读 | 从最下层 Caused by 开始读，找到真正根因 |
| 多数据源调试 | 启动日志里看 DataSource 数量，确认 bean 都创建成功 |
